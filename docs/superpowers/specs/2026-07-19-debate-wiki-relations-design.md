# 辩题 Wiki 与正文引用关系设计

日期：2026-07-19

## 目标

将辩论模块改造成由社区共同维护的 Wiki 式辩题系统：

- 每个辩题是一个稳定的问题节点。
- 标题和正文由登录用户直接协作编辑，并保留完整版本历史。
- 普通讨论服务于正文完善，不参与图谱和结论计算。
- 社区持续投票形成或反转当前结论。
- 辩题间关系只来自正文中的结构化引用。
- 辩论树和关系图是只读投影，不再承担关系编辑。

## 核心原则

1. 系统中不再存在独立的 `Argument` 领域模型。
2. 图中的每个节点都是 `Debate`；一个已有结论的辩题被引用时，才在目标辩题中充当论点。
3. 正文协作、结论投票和关系图谱是三套独立机制，通过稳定的辩题 ID 关联。
4. 结论不是关闭状态。形成结论后仍可编辑正文、讨论、投票和引用。
5. 正文是引用关系的唯一编辑入口，树和图只能读取关系。

## 领域边界

### Debate

`Debate` 表示稳定的核心问题，保存当前标题、当前正文、当前结论和归档状态。

- 标题和正文可以持续完善。
- 编辑不得把核心问题改成另一个问题；核心命题变化时必须新建辩题。
- 发布后的辩题不能通过普通功能硬删除，只能归档。
- 归档保留版本、投票、结论和已有引用，停止普通编辑，并且不能再成为新引用目标。

### DebateRevision

每次保存都生成不可变版本，至少记录：

- 辩题 ID 与递增版本号
- 标题和正文快照
- 编辑者
- 编辑摘要
- 基础版本号
- 该版本解析出的资源引用快照
- 创建时间

用户可以查看版本、比较差异和回退。回退本身生成一个新版本，不改写历史记录。

### Comment

普通讨论继续使用全站通用评论系统，目标为当前辩题。

- 讨论用于建议正文修改、解释编辑理由和处理内容争议。
- 评论和回复不进入辩论树、关系图或结论计算。
- 当前结论变化不会关闭讨论。

### DebateVote

投票直接绑定 `Debate`，不再绑定论点。

- 登录用户一人一票。
- 票的内部枚举为 `yes` 和 `no`，用户可以随时改票；用户界面的方向文案与内部枚举解耦。
- 投票与 Wiki 版本独立，正常文字修订不清空投票。
- 旧的论点投票、结题投票和发起人直接结题流程全部删除。

### DebateConclusionEvent

每次首次形成结论或结论反转都生成不可变事件，记录方向、两侧票数、总票数和时间。

`Debate` 只保存指向当前结论事件的引用，历史事件永久保留。

### DebateReference

`DebateReference` 是正文引用的查询投影，不是独立编辑对象。它至少记录：

- 被引用辩题 ID，即关系来源
- 当前正文所属辩题 ID，即关系目标
- `support` 或 `oppose`
- 产生该投影的目标辩题版本
- 绑定的来源结论事件
- `active`、`stale` 或 `unavailable` 状态

关系方向固定为：

```text
被引用辩题 -> 当前正文所属辩题
```

## 全站资源引用协议

用户提及保持原有形式：

```text
@username
```

普通资源引用使用：

```text
@<type>:<uuid>
```

辩题引用必须增加关系参数：

```text
@debate:<uuid>:support
@debate:<uuid>:oppose
```

首批资源类型采用小写单个英文名词：

| 类型 | 资源 |
| --- | --- |
| `post` | 博客文章 |
| `thread` | 论坛主题 |
| `debate` | 辩题 |
| `feed` | 订阅源 |
| `article` | 订阅文章 |
| `artist` | 艺术家 |
| `album` | 专辑 |
| `song` | 歌曲 |
| `playlist` | 歌单 |
| `podcast` | 播客节目 |
| `episode` | 播客单集 |
| `video` | 视频 |
| `person` | 时间线人物 |
| `event` | 时间线事件 |
| `channel` | 频道 |
| `collection` | 合集 |
| `comment` | 具体讨论内容 |

资源类型是稳定协议，不跟随路由或中文名称变化。只有带冒号的形式才进入资源解析，普通 `@username` 继续走用户提及逻辑。

非辩题资源只渲染链接或资源预览，不进入辩论关系图。

## 编辑与引用同步

### 编辑器行为

- 底层正文保存标准文本标记。
- 编辑器将标记显示为行内引用块，隐藏 UUID，展示资源标题。
- 辩题引用块同时显示关系和来源辩题当前结论。
- 输入资源前缀后打开对应搜索；`@debate:` 只返回已有结论且当前用户可见的辩题。
- 用户仍可直接输入或粘贴完整标记。

### 保存流程

保存必须在后端完成解析和最终校验，不能信任前端解析结果：

1. 检查请求携带的基础版本是否仍是当前版本。
2. 解析正文中的全部资源标记；标题不接受资源引用。
3. 校验资源类型、UUID、访问权限和资源存在性。
4. 校验辩题引用包含合法关系，且来源辩题已有当前结论。
5. 校验同一来源没有同时以支撑和反驳引用当前辩题。
6. 使用保存后的活动关系图检查有向循环。
7. 在同一事务中保存新版本、引用快照和当前关系投影。

任一步失败都回滚整个保存，不能出现正文已更新但图谱未同步的状态。

### 增删规则

- 新增有效标记后，关系立即进入树和图。
- 删除最后一个对应标记后，关系立即退出树和图。
- 任何 Wiki 编辑者都可以删除引用；删除不需要关系投票。
- 同一辩题以相同方向出现多次时允许保存，但图中只生成一条边。
- 同一辩题同时出现 `support` 和 `oppose` 时阻止保存。
- 任何支撑或反驳关系形成有向循环时阻止保存，并返回循环路径。

关系争议通过 Wiki 修改、回退、讨论和必要时页面保护处理，不再提供关系驳回投票。

### 失效与重新确认

来源辩题结论反转时：

1. 所有绑定旧结论事件的活动关系变为 `stale`。
2. 关系立即退出辩论树和关系图。
3. 正文中的引用标记保持不变，并显示“来源结论已变化”。
4. 编辑者可以明确重新确认该引用，使其绑定新的结论事件。
5. 重新确认写入新的引用快照和审计记录，不能由一次无关的正文保存隐式完成。

引用目标后来不可见或不存在时，标记保留并显示不可用状态，对应关系不进入图谱。

新增无效引用会阻止保存；基础版本中已经存在且未修改的失效标记不阻止其他正文修订。

## 结论规则

投票总数至少为 10，并且某一方向票数严格超过总票数的四分之三时，形成该方向的当前结论：

```text
direction_votes * 4 > total_votes * 3
```

- 恰好等于四分之三不形成结论。
- 首次达到条件时生成第一条结论事件。
- 已有结论后，即使原方向跌回四分之三以下，原结论仍保留。
- 只有反方向也严格超过四分之三时，才生成反转事件并替换当前结论。
- 结论计算、历史写入和旧关系失效必须处于同一数据库事务。

## 辩论树与关系图

### 辩论树

- 采用纵向布局。
- 只沿指向当前辩题的 `support` 关系向上展开。
- 首次默认加载 3 层。
- 节点可继续按需展开。
- 由于保存时禁止有向循环，树不会出现循环分支；共享上游节点仍可在不同分支中出现。

### 关系图

- 同时展示 `support` 和 `oppose`。
- 从当前辩题向两个方向查询，首次默认加载 2 跳。
- 节点可继续按需扩展。
- 连线显示关系类型。
- 画布不提供新增、删除或改变关系的操作。

两种视图都只读取 `active` 关系，不显示 `stale` 或 `unavailable` 关系。

## 页面结构

辩题详情页采用聚焦式结构：

- 顶部显示返回入口、辩题标题、当前结论、投票结果和投票操作。
- 主视图使用“正文 / 辩论树 / 关系图”三个标签，一次只展示一个主要内容。
- “编辑 / 版本 / 讨论”作为明确动作打开侧层；移动端使用全屏侧层。
- 正文中的资源引用显示为可读行内块，不直接展示 UUID。
- 失效引用块显示原因和重新确认入口。
- 版本侧层提供历史列表、差异和回退。
- 讨论侧层使用通用评论组件。

页面不展示开发机制说明，错误文案只告诉用户需要修正的引用或版本冲突。

## 并发与争议处理

### 并发编辑

保存请求必须包含当前 `DebateRevision` 的 ID，字段名为 `base_revision`。如果该 ID 已经不是当前版本，服务端拒绝覆盖并返回冲突：

- 前端保留用户草稿。
- 展示基础版本、当前版本和用户草稿之间的差异。
- 用户合并后基于最新版本重新提交。

不采用最后写入覆盖，也不在本阶段引入实时协同编辑。

### 编辑争议

- 普通争议通过版本回退和讨论解决。
- 持续编辑战由管理员临时保护辩题。
- 页面保护期间仍可阅读、投票和讨论，但不能普通编辑正文。
- 核心问题需要改变时创建新辩题，不在原节点上改变投票对象。

## API 与实现边界

后端需要提供以下逻辑能力，具体路由在实施计划中与现有模块风格对齐：

- 获取和更新当前辩题，更新时携带基础版本与编辑摘要。
- 列出版本、读取版本差异和回退版本。
- 设置、修改和读取当前用户的辩题投票。
- 读取投票汇总和结论历史。
- 读取限定深度的辩论树和关系图，并按节点继续展开。
- 重新确认失效引用。
- 归档和临时保护辩题。

以下能力删除：

- `Argument` 创建、编辑、引用、折叠和投票接口。
- 发起人或管理员直接选择结论的接口。
- 投票“要求结题”的接口。
- 独立创建、删除或修改关系的接口。
- 关系驳回投票。

## 数据迁移

- 删除所有旧 `Argument` 数据及其扩展、引用和投票数据。
- 删除旧 `debate_relations` 数据，不迁移到正文。
- 使用现有辩题标题和正文生成版本 1。
- 新关系只能在迁移后通过正文中的 `@debate` 标记重新建立。
- 通用评论表和新的辩题讨论能力保留，但旧 Argument 对应的评论记录不保留。
- 移除旧的 `open / concluded` 关闭语义；当前结论与归档状态分开保存。

## 测试范围

### 后端

- 资源解析：用户提及、普通资源、辩题引用、非法格式和未知类型。
- 引用校验：未结论辩题、相反关系冲突、重复引用和有向循环。
- Wiki 版本：正常保存、回退、基础版本冲突和事务回滚。
- 投票边界：不足 10 票、恰好四分之三、严格超过四分之三、改票和结论反转。
- 结论反转后，旧引用变为失效并退出图谱。
- 辩论树只返回支撑关系和 3 层范围。
- 关系图返回两种关系和 2 跳范围。
- 删除旧 Argument 能力后，通用评论仍能用于辩题讨论。

### 前端

- 资源自动补全和行内引用块。
- 无效、不可用和待重新确认状态。
- 正文、辩论树和关系图标签切换。
- 树与图只读及按需展开。
- 编辑冲突时保留草稿并展示差异。
- 版本历史、回退和讨论侧层。

### 端到端

保留一条覆盖核心闭环的端到端流程：

```text
创建辩题
-> 社区投票形成结论
-> 在另一辩题正文加入引用
-> 引用自动进入树和图
-> 改票使来源结论反转
-> 引用失效并退出图谱
-> 编辑者重新确认后恢复
```

## 非目标

- 不建设通用可编辑图数据库。
- 不在树或关系图中编辑连线。
- 不保留旧 Argument 数据兼容层。
- 不为正文修改增加逐条审核投票。
- 不在本阶段实现实时多人协同编辑。
- 不允许关系形成有向循环。

## 验收标准

1. 用户可以直接编辑辩题 Wiki，查看版本、差异并回退。
2. 普通讨论独立工作，任何结论状态下都可继续讨论。
3. 社区投票按照确定门槛自动形成和反转结论。
4. 只有已有结论的辩题能通过正文标记建立支撑或反驳关系。
5. 正文增删引用后，树和图在同一事务结果中同步变化。
6. 结论反转不会静默沿用旧关系，也不会擅自删除正文标记。
7. 辩论树保持纵向、只展示支撑关系；关系图展示两种关系。
8. 并发编辑不会覆盖其他用户已经保存的新版本。
9. 旧 Argument 和旧关系编辑流程完全移除。
