# Feed 来源发现与管理设计

## 一、目标

为 Atoman 的订阅系统补齐“找源、建源、管源”能力，让 feed 模块从“只能手动填 RSS URL 的阅读功能区”升级为“支持来源发现、RSSHub 建源、后台来源治理的订阅系统”。

本次设计的重点不是重做阅读体验，而是补齐来源层能力，并以最小前台改动接入现有 feed 模块。

## 二、确认后的范围

本次覆盖以下内容：

1. 升级 feed 模块“添加订阅”入口
2. 支持输入网站 URL 自动发现可订阅 feed 候选
3. 接入 RSSHub 作为来源生成能力之一
4. 支持候选 feed 默认自动选择，并允许用户展开切换其他候选
5. 在订阅页增加“全部 / 仅未读”轻量筛选
6. 新增后台可见的订阅源管理能力
7. 支持来源类型标签、基础去重、隐藏来源、来源基础编辑、后台删除
8. 提供简版来源健康状态信息

本次明确不做：

1. 不统一现有主时间线 / 收藏 / 稍后读为单一 view 体系
2. 不新增 list / category，组织方式继续只保留 group
3. 不在前台暴露复杂的来源管理能力
4. 不直接对普通用户暴露 RSSHub 路由、模板、参数等技术概念
5. 不做复杂推荐系统、趋势系统、模板市场
6. 不做高级来源治理动作，如批量治理、复杂重试、禁用、审计流
7. 不做高级智能去重与复杂来源归一

## 三、现状与问题

当前 Atoman feed 模块已经具备：

1. 订阅主页与内容时间线
2. 收藏页、稍后读页、探索页
3. 基础订阅添加能力
4. group 分组能力
5. 已读、收藏、稍后读等基础消费能力

但当前主要问题是：

1. 添加订阅仍然以“手动填写 RSS URL”为主，门槛过高
2. 系统对“来源”缺少统一治理能力，更多只是“订阅后读取内容”
3. 后台缺少正式的来源管理视角
4. 前台用户无法方便地从普通网站地址直接发现可订阅源
5. 未来接入 RSSHub 时，现有模型不足以表达“来源是被生成出来的”这一层语义
6. 订阅页缺少最基本的未读过滤能力

因此，本次工作本质上是把 feed 模块向前补齐到“来源发现与管理”层，而不是重做消费层。

## 四、产品语义

### 4.1 本次要解决什么问题

本次解决的核心问题是：

- 用户关心的是“我要订阅这个网站 / 这个对象 / 这个平台来源”，而不是“我要自己找到一个 RSS URL 再填进去”。

系统应把“找源”和“建源”产品化，而不是把技术细节交给用户。

### 4.2 RSSHub 在系统中的角色

RSSHub 在 Atoman 中是“来源生成能力”，而不是用户直接感知的产品名称。

对普通用户而言，界面应表达为：

- 订阅某个平台对象
- 订阅某个账号、频道、仓库或页面

而不是：

- 手动配置 RSSHub 路由参数

### 4.3 后台与前台职责边界

前台负责：

1. 让用户更容易添加订阅
2. 在订阅页进行轻量阅读筛选

后台负责：

1. 查看来源类型
2. 查看来源状态
3. 编辑来源基础配置
4. 删除来源
5. 隐藏来源
6. 排查与治理异常来源

## 五、设计原则

### 5.1 前台轻，后台重

复杂管理能力优先收敛在后台，不把复杂性暴露到前台用户侧。

### 5.2 先补来源层，不重做消费层

本次不重构现有阅读结构，只在现有订阅页上增加最小必要的“仅未读”筛选。

### 5.3 用户不理解 RSSHub 也能完成订阅

系统应隐藏技术实现细节，优先表达用户目标。

### 5.4 先做基础去重，不做过度智能化

第一阶段先按 feed URL / canonical URL 做基础去重，避免来源脏化，不引入复杂归一算法。

### 5.5 候选默认帮用户选，但保留人工兜底

自动发现到多个 feed 候选时，系统默认给出一个最优候选，同时允许用户展开查看其他候选，避免误判带来的体验问题。

## 六、推荐架构

### 6.1 总体分层

推荐把订阅系统拆成三层理解：

1. 来源发现与生成层
2. 来源管理与治理层
3. 内容消费层

本次重点建设前两层，对第三层仅做最小增强。

### 6.2 来源发现与生成层

支持三种建源方式：

1. 输入网站 URL 自动发现 feed 候选
2. 直接填写 RSS URL
3. 通过 RSSHub provider 生成来源

这三种方式统一收口到“添加订阅”入口中。

### 6.3 来源管理与治理层

后台新增来源管理能力，至少支持：

1. 来源列表
2. 来源类型标签
3. 健康状态与最近同步时间
4. 编辑名称 / group / hidden 状态
5. 删除来源

### 6.4 内容消费层

不重做现有收藏、稍后读、时间线结构。

只在订阅页增加：

1. 全部
2. 仅未读

两个切换状态。

## 七、数据与领域规则

### 7.1 Source

系统中应明确存在“来源对象（Source）”这一层。

Source 至少需要表达：

1. 来源类型
2. 最终 feed 地址
3. 原始站点地址
4. 来源生成方式
5. 是否隐藏
6. 最近同步时间
7. 健康状态
8. 最近失败信息

推荐来源类型至少区分：

1. `rss`
2. `discovered`
3. `rsshub`
4. `internal`

### 7.2 Subscription

Subscription 仍然表达“用户与来源之间的关系”。

本次继续保留 group 作为唯一组织方式。

本次不扩展为 list / category 体系。

### 7.3 Discovery Candidate

自动发现结果应表达为候选对象，至少包含：

1. 标题
2. feed 地址
3. 来源站点地址
4. 候选类型
5. 分数或排序依据
6. 是否默认候选

### 7.4 Hidden 语义

hidden 的产品语义明确为：

1. 该来源继续保留订阅关系
2. 该来源继续同步
3. 该来源默认不进入主订阅时间线

### 7.5 去重规则

第一阶段采用基础去重：

1. 优先按 `canonical_url` 去重
2. 其次按 `feed_url` 去重
3. 创建来源前进行重复检查

如果来源已存在，应优先复用已有来源并建立订阅关系，而不是重复创建新的来源对象。

## 八、前端设计

### 8.1 添加订阅入口

现有添加订阅入口升级为统一建源入口，但不做复杂多步骤向导。

第一阶段至少支持以下三种方式：

1. 网站地址发现
2. 直接填写 RSS
3. 平台对象订阅（底层通过 RSSHub）

### 8.2 自动发现候选交互

当用户输入网站地址后：

1. 系统尝试发现候选 feed
2. 默认自动选中一个最可能的主 feed
3. 用户可以展开查看其他候选
4. 用户确认后完成订阅创建

### 8.3 RSSHub 交互表达

前台不强调 RSSHub 技术术语。

第一阶段只落地一个最小模板：

- 订阅 GitHub 仓库

后续才扩展到更多平台账号、频道或对象。

而不是展示 route / params / template_key 等内部概念。

### 8.4 订阅页未读筛选

在订阅页增加一个轻量未读筛选控件。

第一阶段只支持：

1. 全部
2. 仅未读

控件入口使用用户确认的图标样式。

本次不扩展成复杂筛选器。

## 九、后台设计

### 9.1 后台来源列表

后台需要新增来源管理列表，至少展示：

1. 来源标题
2. 来源类型
3. 所属 group 或相关订阅信息
4. hidden 状态
5. 最近同步时间
6. 健康状态
7. 基础操作入口

### 9.2 后台编辑能力

第一阶段后台仅支持基础编辑：

1. 改名称
2. 改 hidden 状态

`group` 编辑留到后续阶段，不在当前 MVP 中实现。

第一阶段不要求支持复杂建源参数重新编辑。

### 9.3 后台删除能力

后台第一阶段支持删除来源。

第一阶段删除语义明确为：

1. 删除来源时同时删除对应订阅关系
2. 删除来源时删除该来源关联的 `feed_items`
3. 删除来源时同步清理依赖 `feed_items` 的 `feed_item_reads`、`feed_item_stars`、`reading_list_items`
4. 删除后允许通过相同地址重新创建

### 9.4 后台健康状态

第一阶段后台只做简版来源状态展示：

1. 最近同步时间
2. 基本健康状态
3. 最近失败状态或失败摘要

本次不强制要求支持手动重试、重新抓取、禁用等高级操作。

## 十、接口建议

推荐至少补充以下接口能力：

1. 输入网站 URL，返回发现候选
2. 直接用 RSS URL 创建来源并订阅
3. 通过 provider 创建来源并订阅
4. 时间线支持 `unread_only` 过滤
5. 后台来源列表查询
6. 后台来源基础编辑
7. 后台来源删除

本次设计不要求锁死最终 REST 命名，但要求这些能力明确存在。

## 十一、MVP 范围

第一阶段 MVP 只要求做到：

1. 添加订阅入口升级
2. 网站 URL 自动发现
3. RSSHub 最小接入
4. 订阅页“仅未读”筛选
5. 基础去重
6. 后台来源列表
7. 来源类型标签
8. 简版健康状态
9. 来源隐藏
10. 来源基础编辑
11. 后台删除来源

## 十二、验收标准

### 12.1 用户侧

用户应能够：

1. 输入普通网站地址并得到可订阅候选
2. 在默认推荐候选不理想时切换其他候选
3. 继续使用直接 RSS 订阅
4. 在“添加订阅”中完成基于平台对象的来源订阅
5. 在订阅页切换“全部 / 仅未读”

### 12.2 后台侧

后台应能够：

1. 查看来源列表
2. 识别来源类型
3. 查看来源基本状态
4. 编辑来源名称 / group / hidden 状态
5. 删除来源

### 12.3 系统侧

系统应能够：

1. 基础识别重复来源
2. 避免明显重复订阅
3. 让 hidden 来源不进入主订阅流但继续同步

## 十三、风险与约束

1. 网站自动发现结果可能存在误判，因此必须保留候选展开切换能力
2. RSSHub 路由覆盖面很大，第一阶段不宜贪多，应优先选高价值来源模板
3. 后台删除来源的语义如果不先明确，容易引发订阅关系与历史内容的不一致问题
4. 如果过早引入复杂来源编辑和复杂去重，容易超出第一阶段范围

## 十四、结论

本次设计将 Atoman 的 feed 模块从“手动录入 RSS 的消费功能区”升级为“具备来源发现、来源生成与后台来源治理能力的订阅系统”。

落地策略是：

1. 前台保持轻量
2. 复杂治理放后台
3. 先补来源层
4. 不重做阅读结构

这样可以在不推翻现有 feed 消费体验的前提下，把 Atoman 向更成熟的 Folo 式订阅系统推进一大步。
