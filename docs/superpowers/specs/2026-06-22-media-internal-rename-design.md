# 内容模块内部重命名设计

## 背景

内容模块当前对外名称已经是“内容”，正式域名已经是 `media.atoman.org`。但代码内部仍然大量使用 `kanbo` 作为模块技术键、目录名、composable 名、测试名和部分后端保留名。

这带来三个实际问题：

1. 对外域名与对内模块键完全脱节。
2. 新功能开发需要持续在“内容”“media”“kanbo”之间做映射，增加认知负担。
3. 路由、目录、权限、测试、站点上下文的命名不一致，会持续制造维护噪音。

本次重命名不是文案修正，而是运行时语义统一。目标不是把 `kanbo` 换成 `content` 再多留一层映射，而是让 `media` 成为唯一的主技术语义。

## 目标

阶段 A 的目标如下：

1. 前端运行时主模块键从 `kanbo` 切换到 `media`。
2. 前端核心目录、视图、组件、composable、测试命名同步改为 `media` 语义。
3. 后端权限、站点保留名、命名空间主语义同步改为 `media`。
4. 不保留 `kanbo` 作为兼容模块入口。
5. 对外和对内统一以 `media` 为唯一主语义。

## 非目标

阶段 A 不做以下事情：

1. 不清理全部历史设计文档、旧计划文档、review 文档中的 `kanbo` 文本残留。
2. 不要求一次性重写所有历史测试描述和历史文件名之外的非运行时材料。
3. 不调整内容模块首页的信息架构和视觉方案。
4. 不保留 `kanbo` 兼容入口。
5. 不做后端业务 API 结构重构，只处理模块键、命名空间、权限键和命名统一。

这些工作之外的历史债清理属于阶段 B。

## 设计原则

### 1. 唯一语义

运行时主语义只有一个：`media`。

这意味着：

1. 新代码、新目录、新测试、新模块键全部以 `media` 为准。
2. `kanbo` 不再作为模块别名存在。
3. `content` 也不作为新的中间技术键引入。

### 2. 先统一运行时，再清历史债

优先级应是：

1. 路由
2. 站点上下文解析
3. URL 生成
4. 目录与模块命名
5. 权限与保留名
6. 核心测试

历史文档、旧计划和 review 文本清理后置，避免和运行时重构混做。

### 3. 对外入口与对内语义一致

以下行为在阶段 A 之后必须成立：

1. `media.atoman.org` 对应 `media` 模块
2. 本地开发入口使用 `?site=media`
3. 生成内容模块链接时始终生成 `media` 对应 URL

以下旧入口将不再保留：

1. `kanbo.atoman.org`
2. `?site=kanbo`

## 前端设计

### 模块键

`ModuleRoomKey` 的内容模块主键应切换到 `media`。

结果应为：

1. 导航、站点上下文、模块可见性、权限读取等主路径全部以 `media` 为准。
2. 原 `kanbo` 从前端主模块键集合中移除。

### 路由表

`moduleRoutes` 中原 `kanbo` 路由表应改为 `media`。

要求：

1. 内容模块所有根路径和子路径继续保持当前 URL 结构不变。
2. 只是模块表键和值班逻辑切换，不改变 `media.atoman.org/articles`、`/videos`、`/podcasts`、`/create` 等现有路径。

### 站点上下文解析

`siteContext` 应变成：

1. `media` -> `media`
2. 本地 `site=media` -> `media`

不再支持：

1. `kanbo` -> 模块
2. `site=kanbo` -> 模块

一旦阶段 A 落地，`kanbo` 应被视为普通非模块子域名，除非后端保留名策略另有需要。

### URL 生成

`siteUrls` 的模块 URL 生成应以 `media` 为输入主键，正式输出仍是：

1. `media` 模块 => `media.atoman.org`

不再保留旧模块别名映射。

### 目录与命名

以下前端命名需要统一切到 `media`：

1. `src/views/kanbo` -> `src/views/media`
2. `src/components/kanbo` -> `src/components/media`
3. `useMedia*` -> `useMedia*`
4. `Media*View` / `Media*Section` / `Media*Workspace` -> `Media*`

要求：

1. 目录、文件名、import 路径、测试 import 必须一致更新。
2. 不保留“新模块还住在旧目录”这种中间态。

### 测试

核心测试需要同步到 `media` 语义，至少覆盖：

1. 路由键切换后内容模块仍可用
2. `media.atoman.org` 能落到 `media`
3. 本地 `site=media` 能落到 `media`
4. 内容首页、文章页、播客页、视频页、创作页仍然工作
5. 旧 `kanbo` 测试文件名和 describe 文案改成 `media`

## 后端设计

### 站点保留名

后端保留站点名需要确保以下值被保留：

1. `media`
2. `kanbo`

原因：

1. `media` 防止用户或频道抢占正式技术键与子域名
2. `kanbo` 即便不再作为模块入口，也不应允许被用户或频道抢占，避免历史链接或认知冲突

### 站点命名空间解析

后端若在模块保留列表、命名空间服务、站点访问默认模块键里显式列出 `kanbo`，应切换为：

1. `media` 作为主语义
2. `kanbo` 仅作为保留值，不作为模块别名返回

要求：

1. 新返回值、权限键和默认模块配置以 `media` 为主
2. 不把 `kanbo` 当成可继续使用的模块键

### 权限与站点访问

若站点访问配置当前以 `kanbo` 作为模块键，需要改成 `media`。

要求：

1. 新配置、验证逻辑、默认配置主键改为 `media`
2. 若已有持久化数据记录 `kanbo`，则需要迁移或兼容读取

如果后端已有持久化数据结构会把模块键写入数据库，则阶段 A 里必须明确迁移或兼容策略，不能只改类型定义。

## 风险与控制

### 风险 1：改动面大

前端当前命中约 396 处，后端命中约 7 处。若直接全量文本替换，极容易把历史文档和运行时代码混在一起，导致 review 困难。

控制方式：

1. 阶段 A 只处理运行时代码、核心测试和必要的目录重命名
2. 阶段 B 再清历史文本债

### 风险 2：旧入口失效

这次设计明确不保留：

1. `kanbo.atoman.org`
2. `?site=kanbo`

这不是副作用，而是设计决策。

控制方式：

1. 提前在实现中移除旧解析逻辑
2. 用测试和浏览器验证确认 `media` 唯一生效
3. 在上线前明确通知依赖旧入口的开发和运营使用方

### 风险 3：配置或权限数据失配

如果后端权限模块键从 `kanbo` 改到 `media`，而已有配置仍旧持有 `kanbo`，可能导致内容模块可见性或功能开关失配。

控制方式：

1. 明确检查是否有数据库或配置落盘依赖
2. 若有，则增加迁移或兼容读取逻辑
3. 不允许只改 TypeScript / Go 常量而不验证读取路径

## 分阶段范围

### 阶段 A：运行时语义重构

包含：

1. 前端模块键与路由键改为 `media`
2. 前端目录、组件、view、composable、核心测试改名
3. 前端 `siteContext` / `siteUrls` 改为仅支持 `media`
4. 后端保留名、权限模块键、命名空间主语义改为 `media`
5. 线上入口验证

### 阶段 B：历史债清理

包含：

1. 历史 spec / plan / review 文档批量改名
2. 剩余测试文件名、注释、describe、计划文件残留清理

## 验证要求

阶段 A 完成后至少需要验证：

1. `media.atoman.org` 能进入内容模块
2. 本地 `?site=media` 生效
3. 内容首页、文章、播客、视频、创作路由正常
4. 内容模块权限和可见性未回归
5. 旧数据或默认配置不会因为模块键切换而失配
6. `kanbo.atoman.org` 和 `?site=kanbo` 不再被识别为内容模块入口

## 结论

阶段 A 不应做成一次性“全仓文本替换”，而应做成一次 `media` 唯一语义的运行时重构。

最终决策如下：

1. 主技术键从 `kanbo` 切到 `media`
2. 不保留 `kanbo` 作为模块兼容入口
3. 正式域名继续使用 `media.atoman.org`
4. 前端目录、视图、组件、composable、核心测试改到 `media` 语义
5. 后端权限与站点命名空间主语义同步改到 `media`
6. 历史文档和剩余文本债延后到阶段 B 清理
