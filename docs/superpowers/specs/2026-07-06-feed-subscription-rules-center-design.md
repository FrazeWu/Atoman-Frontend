# 订阅规则中心设计

## 目标

为订阅模块增加一个后端托管的“订阅规则中心”，让用户可以用规则统一整理新增订阅和已有订阅，而不是逐个手工设置。

第一版目标：

- 支持创建、编辑、启用、停用、删除、排序订阅规则。
- 支持对新增订阅自动套用规则。
- 支持对已有订阅批量重算并落库。
- 规则条件只支持来源分类、指定来源、关键词。
- 规则动作只支持分组、静音、自动已读、自动稍后阅读。

第一版不做：

- 条件组合器（且 / 或）
- 拖拽排序
- dry-run 预览
- 应用历史列表
- 异步任务系统
- 删除订阅、改标题、健康检查等非整理类动作

## 现状

当前订阅管理能力已经存在，但比较分散：

- 前端 [feed.ts](/Users/fafa/projects/Atoman/Atoman-Frontend/src/stores/feed.ts:136) 维护了本地 `filterRules` 和 `automationRules`。
- [SubscriptionManageSheet.vue](/Users/fafa/projects/Atoman/Atoman-Frontend/src/components/feed/SubscriptionManageSheet.vue:1) 已经提供分组、静音、自动已读、自动稍后阅读、关键词隐藏等单点能力。
- [FeedView.vue](/Users/fafa/projects/Atoman/Atoman-Frontend/src/views/feed/FeedView.vue:344) 在时间线渲染阶段消费这些本地规则。

这套方式能工作，但有三个问题：

1. 规则只在当前设备本地生效，不能跨端同步。
2. 规则不能正式应用到已有订阅配置，只是前端渲染时临时生效。
3. “新增订阅自动整理”和“已有订阅批量重整”没有统一后端入口。

## 设计原则

- 保持简单，不做通用规则引擎。
- 规则是正式业务数据，存后端，不再依赖本地存储。
- 规则执行也在后端，前端只负责管理界面和触发动作。
- 新增订阅自动应用与已有订阅批量重算必须复用同一套计算逻辑。
- 规则只改托管字段，不碰订阅标题等人工编辑字段。

## 用户模型

用户管理的不是单个订阅源，而是“订阅整理方式”。

第一版核心体验：

1. 用户在订阅管理面板中创建一条规则。
2. 保存成功后，可选择立即应用到已有订阅。
3. 后续新增订阅时，系统自动套用当前启用规则。
4. 用户也可以单独重跑某条规则涉及的订阅，或重算全部订阅。

## 规则模型

每条规则包含：

- `name`：规则名称
- `enabled`：是否启用
- `position`：排序位置
- `match_type`：匹配类型
- `conditions_json`：匹配条件
- `action_group_id`：目标分组，可空
- `action_muted`：是否静音，可空
- `action_auto_mark_read`：是否自动已读，可空
- `action_auto_add_reading_list`：是否自动稍后阅读，可空

### 匹配类型

第一版只支持三种：

1. `source_category`
   条件示例：
   ```json
   { "categories": ["podcast", "news"] }
   ```

2. `source_ids`
   条件示例：
   ```json
   { "source_ids": ["source-a", "source-b"] }
   ```

3. `keywords`
   条件示例：
   ```json
   { "keywords": ["AI", "招聘"] }
   ```

### 关键词范围

第一版关键词只匹配：

- 订阅标题
- 来源标题

不匹配文章标题、摘要、正文，避免把“订阅整理规则”和“内容过滤规则”混在一起。

## 冲突与覆盖规则

规则按列表顺序执行，后面的覆盖前面的。

作用到同一个订阅时：

- 分组：最后命中的 `action_group_id` 生效。
- 静音：最后命中的 `action_muted` 生效。
- 自动已读：最后命中的 `action_auto_mark_read` 生效。
- 自动稍后阅读：最后命中的 `action_auto_add_reading_list` 生效。

这是第一版唯一排序语义，不再引入额外优先级系统。

## 规则执行模型

后端统一提供一套规则重算逻辑，供两种场景复用：

1. 新增订阅后自动应用
2. 对已有订阅批量重算

执行流程：

1. 取出当前用户全部启用规则，并按 `position` 排序。
2. 对目标订阅集合逐个计算命中规则。
3. 从前到后合并动作，后命中的覆盖前面的结果。
4. 将最终结果一次性写回订阅托管字段。

注意：

- “应用某条规则”在后端语义上不是只执行这一条。
- 它表示“以当前整套启用规则为准，对这条规则影响的目标订阅做完整重算”。

## 托管字段边界

规则系统第一版只负责以下字段：

- 订阅分组
- 订阅静音
- 自动已读
- 自动稍后阅读

不负责：

- 订阅标题
- OPML 导入结果
- 健康检查状态
- 内容级收藏 / 稍后阅读条目

如果当前“静音 / 自动已读 / 自动稍后阅读”仍有部分逻辑只存在前端本地，需要在实现时收口到后端正式字段，否则规则中心只能做到半套。

## 数据存储

### 新增表

建议在后端新增 `feed_subscription_rules` 表：

- `id`
- `user_id`
- `name`
- `enabled`
- `position`
- `match_type`
- `conditions_json`
- `action_group_id`
- `action_muted`
- `action_auto_mark_read`
- `action_auto_add_reading_list`
- `created_at`
- `updated_at`

### 设计取舍

第一版不拆独立条件子表，直接用 `conditions_json` 承载条件：

- 规则类型目前只有三种，JSON 足够简单。
- 后续新增 `domain`、`group` 等条件也能平滑扩展。
- 可以避免把第一版过早做成规则 DSL。

## 后端接口

第一版新增以下接口：

1. `GET /api/v1/feed/subscription-rules`
   获取当前用户规则列表，按 `position` 返回。

2. `POST /api/v1/feed/subscription-rules`
   创建规则。

3. `PUT /api/v1/feed/subscription-rules/:id`
   更新规则。

4. `DELETE /api/v1/feed/subscription-rules/:id`
   删除规则。

5. `PUT /api/v1/feed/subscription-rules/reorder`
   一次提交完整顺序数组，后端整体更新排序。

6. `POST /api/v1/feed/subscription-rules/apply`
   统一触发规则应用。

### `apply` 请求模式

支持两种请求：

- 应用某条规则影响范围：
  ```json
  { "rule_id": "rule-1" }
  ```

- 重算全部订阅：
  ```json
  { "all": true }
  ```

后端虽然接收不同模式，但都走同一套完整重算逻辑。

### `apply` 响应

返回简短结果摘要：

- `scanned_count`
- `updated_count`
- `group_changed_count`
- `muted_changed_count`
- `auto_mark_read_changed_count`
- `auto_add_reading_list_changed_count`

前端用它展示最近一次应用结果。

## 前端结构

第一版不新增独立规则页面，直接挂在 [SubscriptionManageSheet.vue](/Users/fafa/projects/Atoman/Atoman-Frontend/src/components/feed/SubscriptionManageSheet.vue:1) 内。

### 面板结构

1. 顶部工具栏
   - 新建规则
   - 重算全部订阅
   - 最近一次应用结果摘要

2. 规则列表区
   每条规则卡片显示：
   - 名称
   - 条件摘要
   - 动作摘要
   - 启用 / 停用
   - 上移
   - 下移
   - 编辑
   - 应用到已有订阅
   - 删除

3. 规则编辑 Sheet
   字段包括：
   - 规则名称
   - 匹配类型
   - 条件输入
   - 动作输入
   - 启用状态
   - 保存按钮

### 保存后的交互

创建或编辑规则成功后：

- 弹确认：“是否立即应用到已有订阅？”
- 用户确认后调用 `apply`

规则列表中也保留“应用到已有订阅”按钮，供后续重跑。

### 排序交互

第一版只做“上移 / 下移”按钮，不做拖拽。

原因：

- 实现更稳
- 顺序语义已经足够明确
- 能避免移动端拖拽交互额外成本

## 本地规则迁移策略

当前前端已有本地：

- `filterRules`
- `automationRules`

第一版上线后，针对规则中心覆盖的 4 类托管字段，应以“后端规则 + 后端订阅字段”为准，不再长期依赖本地规则作为主逻辑。

建议迁移顺序：

1. 先补齐后端托管字段与规则系统。
2. 前端切换到消费后端返回的订阅整理状态。
3. 再逐步移除或降级本地规则入口，避免双轨并存。

关键词隐藏如果暂时还保留本地能力，必须在 UI 上明确区分它不属于第一版订阅规则中心的后端托管能力。

## 实现顺序

1. 后端收口订阅托管字段
2. 后端新增规则表与 CRUD
3. 后端实现统一规则重算逻辑与 `apply`
4. 新增订阅流程接入规则自动应用
5. 前端新增规则列表、编辑 Sheet、排序和应用入口
6. 前端逐步替换现有本地整理逻辑

## 主要风险

### 本地规则与后端规则并存

如果两套规则长期同时生效，用户会无法判断最终结果来源。

结论：

- 第一版必须明确“订阅整理字段以后端规则结果为准”。

### 关键词语义扩散

如果关键词匹配范围扩到文章内容层，会把订阅规则和内容过滤搅在一起。

结论：

- 第一版关键词只匹配订阅标题和来源标题。

### “应用某条规则”的认知歧义

用户可能误以为点击某条规则的“应用”按钮时，只执行单条规则。

结论：

- 前端文案要明确为“按当前规则顺序重新计算相关订阅”。

## 验收标准

- 用户可以创建、编辑、启用、停用、删除规则。
- 用户可以调整规则顺序，后面的规则覆盖前面的。
- 新增订阅后能自动获得规则结果。
- 用户可以对已有订阅执行规则重算。
- 重算结果会正式写入订阅托管字段，而不是只在前端临时显示。
- 前端可以看到一次规则应用的摘要结果。
