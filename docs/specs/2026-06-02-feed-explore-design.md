# Feed Explore 页面设计

## 一、目标

为 `http://localhost:5173/explore?site=feed` 提供实际可用的“探索”能力。

本次目标不是做订阅源管理，也不是做站内博客聚合，而是让 feed 模块拥有一个真正可用的“全站外部内容发现流”。

用户进入该页面后，应能看到来自全站外部 RSS / 播客源的内容列表，并可在“随机”与“热门”两种模式之间切换。

## 二、确认后的范围

本次只覆盖以下内容：

1. `feed` 模块的 `/explore` 页面
2. 外部 RSS / 播客条目的探索流
3. `random` 与 `popular` 两种排序模式
4. 前端失败态与空态收口
5. 对应的后端查询接口

本次明确不做：

1. 站内博客文章聚合
2. 用户、频道、合集级订阅推荐
3. 基于用户行为的真实热度系统
4. 复杂推荐算法
5. 新的管理后台配置

## 三、现状与问题

当前 feed 模块路由已经把 `/explore` 指向 [FeedRecommendedView.vue](/Users/fafa/Downloads/projects/Atoman/web/src/views/feed/FeedRecommendedView.vue:1)。

该页面已经具备：

1. 标题区与“随机 / 热门 / 返回订阅”操作
2. `feed_item` 卡片渲染
3. 收藏、稍后阅读、播客播放、原文跳转等交互
4. 对 `GET /feed/explore/feed?sort=...` 的前端请求

但当前问题是：

1. 页面数据源没有真正跑通
2. 页面失败时只会静默为空
3. “探索”语义尚未落实为真实的全站内容流

因此本次工作本质上是补齐这个页面所依赖的后端能力，并把前端收口到可用状态。

## 四、产品语义

### 4.1 探索页是什么

Feed Explore 页面表达的是：

- “从全站已经接入的外部订阅内容里，发现值得看的条目”

而不是：

- “管理我的订阅”
- “推荐我去关注哪些源”
- “把我自己的订阅时间线再展示一遍”

因此该页应返回全局外部内容，而不是当前用户自己的订阅内容。

### 4.2 内容范围

本次探索页只展示：

- `external_rss` 来源对应的 `feed_items`

不展示：

1. 站内 `post`
2. `internal_user`
3. `internal_channel`
4. `internal_collection`

### 4.3 排序语义

`random`：

- 从符合条件的全局外部 `feed_items` 中随机抽取一批结果

`popular`：

- 不是社交热度
- 本次定义为“近期优质内容优先”
- 通过简单内容完整度信号与发布时间共同排序

## 五、设计原则

### 5.1 最小改动

优先复用现有页面 [FeedRecommendedView.vue](/Users/fafa/Downloads/projects/Atoman/web/src/views/feed/FeedRecommendedView.vue:1) 的结构与卡片样式，不重做信息架构。

### 5.2 先可用，再复杂

本次只实现一个简单可靠的探索流，不引入推荐系统、行为打分、个性化召回等复杂能力。

### 5.3 数据质量优先于“看起来聪明”

与其做一个名义上叫“热门”但实际不可解释的排序，不如明确提供“近期优质内容”。

## 六、推荐架构

### 6.1 前端

继续使用 [FeedRecommendedView.vue](/Users/fafa/Downloads/projects/Atoman/web/src/views/feed/FeedRecommendedView.vue:1) 作为页面主体。

前端职责：

1. 根据 `sort` 请求探索接口
2. 渲染返回的 `feed_item` 列表
3. 提供加载中、空态、失败态
4. 复用现有的收藏、稍后阅读、播客播放、原文跳转交互

### 6.2 后端

在现有 `feed` 模块下补充 explore HTTP 接口与查询逻辑。

推荐接口：

- `GET /api/v1/feed/explore/feed?sort=random|popular&limit=20`

接口返回格式尽量复用当前 feed 页面已使用的数据结构：

```json
{
  "data": [
    {
      "type": "feed_item",
      "feed_item": {}
    }
  ]
}
```

这样前端无需引入新的视图模型。

### 6.3 查询分层

推荐职责分布：

1. HTTP handler 负责参数解析与响应
2. feed service 负责排序模式分发
3. repo / 查询层负责实际 SQL 查询

不新增新的跨模块抽象。

## 七、数据规则

### 7.1 基础筛选

进入探索流的记录必须满足：

1. 对应 `feed_source.source_type = external_rss`
2. `feed_items.title` 非空
3. `feed_items.link` 非空

这三个条件是最小可用门槛。

### 7.2 内容完整度信号

`popular` 模式采用简单加权，优先展示更完整的内容。

可用信号：

1. 有 `published_at`
2. 有 `summary`
3. 有 `image_url`
4. 有 `enclosure_url`

这不是长期推荐算法，只是为了让第一页结果更像“能看”的内容。

### 7.3 去重策略

不做复杂去重。

最小策略：

1. 相同 `link` 的重复项只保留一条
2. 优先保留发布时间更近、字段更完整的那条

如果现有 SQL 查询层实现这个去重成本偏高，也可以先只依赖数据库现状，不额外做复杂 dedupe，只要最终结果不明显重复即可。

## 八、排序定义

### 8.1 random

`random` 模式要求：

1. 从符合条件的外部 `feed_items` 中随机取 `limit` 条
2. 允许不同次请求返回重复结果
3. 不要求稳定分页

本模式重点是“发现感”，不是精确列表浏览。

### 8.2 popular

`popular` 模式要求：

1. 先按内容完整度信号进行简单评分
2. 再按 `published_at` 倒序
3. 结果应明显优于纯随机列表的可读性

推荐思路：

1. 有发布时间优先
2. 有摘要优先
3. 有封面优先
4. 有播客附件优先
5. 同分时按发布时间倒序

不需要把分数系统抽象成可配置规则。

## 九、前端行为

页面保留当前操作区：

1. `随机`
2. `热门`
3. `返回订阅`

页面状态定义：

1. 请求中：显示 skeleton
2. 请求成功但为空：显示“暂无发现内容”
3. 请求失败：显示“加载失败，请稍后再试”

页面卡片继续沿用当前 `feed_item` 呈现：

1. 标题
2. 摘要
3. 来源标题
4. 发布时间
5. 收藏
6. 稍后阅读
7. 播客播放（如果有）
8. 原文跳转

本次不新增新的复杂筛选器。

## 十、错误处理

后端：

1. `sort` 只接受 `random` 或 `popular`
2. 其他值返回 `400`
3. 查询异常返回现有风格错误

前端：

1. 请求失败时不静默吞掉错误
2. 页面应明确展示失败态
3. 失败后用户可通过再次切换 tab 触发重试

## 十一、测试策略

本次只做最小必要验证。

后端至少验证：

1. `sort=random` 能返回外部 `feed_item`
2. `sort=popular` 能返回外部 `feed_item`
3. 非法 `sort` 返回 `400`

前端至少验证：

1. `/explore?site=feed` 能显示内容流
2. `随机 / 热门` 切换会触发不同请求
3. 失败态不再表现为空白

工程验证：

1. 前端改动后运行 `bun run type-check`
2. 后端改动后运行 `go build ./...`

## 十二、不做的事

本次明确不做：

1. 站内博客内容并入 feed explore
2. 用户行为热度统计
3. 个性化推荐
4. 管理端推荐位
5. 复杂缓存
6. 数据库结构变更

## 十三、结论

本次将 `feed` 模块的 `/explore` 页面落实为“全站外部内容发现流”。

实现方式是：

1. 复用现有前端页面结构
2. 在 `feed` 模块补齐 explore 接口
3. `random` 提供随机发现
4. `popular` 提供近期优质内容排序

这样可以用最小改动，把当前空壳页面变成一个真正可用的 feed 探索入口。
