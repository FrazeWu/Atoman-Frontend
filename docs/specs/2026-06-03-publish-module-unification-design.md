# 刊播模块整合设计

## 一、目标

将现有博客、播客、视频三个一级模块整合为一个新的统一功能模块：`刊播`。

本次整合不仅覆盖 Studio / 后台管理体验，也覆盖前台用户可见入口、模块命名、导航结构与频道内工作流。目标不是把三类内容强行做成同一种数据模型，而是在保留文章、播客、视频各自专业能力的前提下，提供统一的内容生产、整理、浏览与关系能力入口。

## 二、确认后的范围

本次覆盖以下内容：

1. 顶层导航中 blog / podcast / video 的统一收敛
2. 新的一级模块命名、页面标题与侧边栏命名
3. 频道上下文、合集容器与创作工作流的统一重构
4. 刊播首页（创作页）信息架构重做
5. 文章 / 播客 / 视频三类内容的统一组织方式
6. 订阅 / 收藏入口并入刊播模块后的角色定义
7. 对应前后端视图模型与路由语义的调整方向

本次明确不做：

1. 音乐模块并入刊播
2. 一张统一的内容总表替代 post / podcast / video
3. 用一套统一编辑器替代文章、播客、视频现有编辑能力
4. 跨频道总览后台
5. 新的推荐算法或复杂内容分析体系
6. 为保持统一而抹平各内容类型的专业能力差异

## 三、命名与导航决策

### 3.1 顶层功能命名

全站顶层导航统一使用功能名，不再使用房间名。

这意味着：

- topbar 使用功能名
- 具体页面标题使用功能名
- 模块内部侧边栏使用功能名
- 不再在用户可见层使用山门、法堂、静阁、影壁等房间式名称

### 3.2 合并后的统一模块名

博客、播客、视频合并后的用户可见一级模块名为：

- `刊播`

命名理由：

1. `刊` 对应文章、编修、刊发
2. `播` 对应播客、视频、传播
3. 相较于“媒体”，`刊播` 不会自然把音乐也包含进去
4. 相较于“发布”或“创作”，`刊播` 更能同时覆盖浏览、订阅、收藏等围绕内容展开的能力

### 3.3 Topbar 顺序

确认后的 topbar 前半段顺序固定为：

1. 订阅
2. 刊播
3. 音乐
4. 论坛

后续模块顺序可灵活安排。

## 四、刊播模块的产品定位

刊播不是传统博客后台，也不是泛化的媒体中心，而是一个统一承载文章、播客、视频内容的功能模块。

它统一的是：

1. 一级模块认知
2. 导航入口
3. 频道内工作流
4. 创作工作台壳
5. 内容总览与关系能力入口

它保留差异的是：

1. 内容类型
2. 合集类型
3. 编辑器形态
4. 视频与播客特有的专业能力
5. 各内容类型更适合的展示方式

## 五、核心对象模型边界

### 5.1 频道（Channel）

频道是刊播模块的经营上下文。

职责：

1. 决定用户当前正在经营哪个内容空间
2. 通过 topbar 中的频道选择器切换当前可经营频道
3. 刊播模块内的创作页默认受当前频道约束；文章 / 播客 / 视频类型页是全站探索视角

规则：

1. 进入刊播的创作页后，必须处于某个当前频道上下文
2. 创作页不提供跨频道总览
3. 切换频道后，创作页回到新频道的总览态
4. topbar 频道选择器只展示当前登录用户可经营的频道，不展示全站频道

### 5.2 合集（Collection）

合集是创作容器，而不是普通标签或纯分组手段。

职责：

1. 作为发布内容前必须进入的工作上下文
2. 决定当前创作行为属于哪一类内容
3. 承载进入具体工作态的分流作用

规则：

1. 必须先选中合集，才能发布内容
2. 一个合集只能承载一种内容类型
3. 合集类型必须显式区分为：`article | podcast | video`

### 5.3 内容实体

底层不统一为一张总内容表，继续保留独立内容实体：

1. 文章（Post）
2. 播客（PodcastEpisode 或现有播客实体）
3. 视频（Video）

统一发生在模块组织层与视图层，不发生在底层存储层。

### 5.4 统一媒体项视图

为了支持刊播模块的总览、筛选、列表与工作台，需要引入一个统一的上层视图模型，用于把不同类型内容组织到同一套模块能力中。

统一视图字段至少包括：

- `id`
- `type`（article / podcast / video）
- `title`
- `channel_id`
- `collection_id` 或 `collections`
- `cover` / `thumbnail`
- `summary`
- `status`
- `published_at`
- `updated_at`
- `owner`
- `url`
- `edit_url`

同时允许各类型保留扩展字段：

文章扩展示例：

- 摘要
- 字数
- 评论状态

播客扩展示例：

- 音频地址
- 时长
- 季 / 集信息
- shownotes 状态

视频扩展示例：

- 封面
- 时长
- 上传 / 转码状态
- 时间戳互动信息

这样可以统一组织，而不会把不同类型内容硬压成完全相同的对象。

## 六、刊播模块的页面结构

### 6.1 模块主入口

点击 topbar 中的 `刊播` 后，默认进入刊播模块中的：

- `创作` 页

`创作` 是刊播模块的统一工作台主页，而不是附属功能页。

### 6.2 模块内部侧边栏

刊播模块内部固定提供以下功能入口：

1. 文章
2. 视频
3. 播客
4. 订阅
5. 收藏
6. 创作

这六项不是同一语义层，但共同构成刊播模块的完整能力面：

- 内容视角：文章 / 视频 / 播客
- 关系能力：订阅 / 收藏
- 工作台入口：创作

### 6.3 页面标题

用户可见页面标题全部使用功能名，例如：

- 刊播
- 创作
- 文章
- 视频
- 播客
- 订阅
- 收藏

不再在页面标题层使用房间式命名。

## 七、创作页设计

### 7.1 创作页是刊播模块主页

创作页承担统一工作台角色，默认展示当前频道的总览态，并作为进入具体合集工作态的入口。

它是刊播模块中最重要的页面，不应被视为次级功能入口。

### 7.2 频道总览态

进入创作页后，默认先进入当前频道的总览态。

总览态分三段：

#### 上方：合集层

展示当前频道下的合集，承担以下职责：

1. 快速进入具体工作容器
2. 告知合集类型
3. 呈现少量必要状态

推荐展示字段：

1. 合集名称
2. 合集类型（文章 / 播客 / 视频）
3. 内容数量
4. 最近更新时间
5. 必要时的草稿或未完成提示

合集层优先服务“快速进入工作”，不做重仪表盘。

#### 中部：文章 + 播客流

文章与播客在频道总览态下共用一个线性列表。

规则：

1. 默认只显示 5 条
2. 提供“展开更多”入口
3. 仅展示当前频道范围内的内容
4. 用于表现最近更新、近期工作动态与连续内容流

文章与播客可以同列表展示，但它们不共用合集。

#### 下方：视频区

视频在频道总览态下单独以卡片形式呈现。

规则：

1. 不混入文章 / 播客线性列表
2. 保持视觉资产与作品单元的卡片化表达
3. 仅展示当前频道范围内的视频内容

这使刊播首页不是“假统一列表”，而是根据媒介自然形态组织内容。

### 7.3 合集工作态

当用户在合集层中选中某个合集时，创作页从频道总览态切换到合集工作态。

规则：

1. 选中合集后进入对应工作模式
2. 发布入口激活
3. 工作视图由合集类型决定

具体分流：

#### 选中文章合集

- 进入文章工作视图
- 以线性内容管理、创建和编辑为主

#### 选中播客合集

- 进入播客工作视图
- 以单集管理与发布为主

#### 选中视频合集

- 进入视频工作视图
- 以卡片式视频管理与上传编辑为主

### 7.4 发布规则

发布必须严格依赖已选中的合集。

规则：

1. 未选中合集前，发布按钮不可用
2. 若用户尝试发布，应明确提示先选择合集
3. 选中合集后，发布类型自动由合集类型决定
4. 发布流程中不应再次让用户二次选择内容类型

这套约束可以显著减少歧义与错误路径。

## 八、内容视角页设计

### 8.1 文章页

文章页是本网站全部文章的探索视角页，语义参考订阅模块的探索页。

职责：

1. 浏览本网站范围内的全部公开文章
2. 默认按最新探索；热门排序只有在后端提供真实统计排序后才展示
3. 进入文章详情或编辑
4. 可返回创作页继续频道工作台操作

### 8.2 播客页

播客页是本网站全部播客单集的探索视角页，语义参考订阅模块的探索页。

职责：

1. 浏览本网站范围内的全部公开播客单集
2. 默认按最新探索；随机或热门排序只有在后端提供真实排序能力后才展示
3. 进入播客详情或编辑
4. 可返回创作页继续频道工作台操作

### 8.3 视频页

视频页是本网站全部视频的探索视角页，语义参考订阅模块的探索页。

职责：

1. 浏览本网站范围内的全部公开视频
2. 支持最新 / 热门排序
3. 使用更适合视频内容的卡片式或视觉探索视图
4. 可返回创作页继续频道工作台操作

### 8.4 类型页与创作页的边界

创作页与类型页职责必须明确区分：

创作页负责：

1. 统一入口
2. 频道总览
3. 选择合集
4. 进入工作态
5. 开始发布

类型页负责：

1. 按媒介探索本网站全部公开内容
2. 做更完整的发现、筛选和浏览
3. 从类型角度进入详情，并可回到创作页继续工作台操作

这样不会把所有职责都堆进创作页，也不会让类型页与工作台重复。

## 九、订阅与收藏的角色定义

### 9.1 全局能力定位

订阅与收藏入口放在刊播模块内部，但逻辑上属于用户全局能力，而不是频道经营对象。

规则：

1. 订阅不受当前频道切换影响
2. 收藏不受当前频道切换影响
3. 频道切换时，这两页数据结果不变化

### 9.2 订阅页

订阅页用于统一查看用户全站范围内关注的内容更新。

页面副标题或页面提示应明确说明：

- 此页展示的是全局订阅内容，不限定当前频道

### 9.3 收藏页

收藏页用于统一查看用户全站范围内保存的内容。

页面副标题或页面提示应明确说明：

- 此页展示的是全局收藏内容，不限定当前频道

### 9.4 视觉区分要求

为了避免用户误认为订阅 / 收藏是频道内页，界面上至少应明确：

1. 标题或副标题说明其全局属性
2. 频道切换器保持可见，但不驱动此页结果变化
3. 必要时提供轻量全局提示，例如“全局内容”

## 十、路由与状态语义

### 10.1 顶层路由原则

用户应感知为进入一个统一模块，而不是 blog / podcast / video 三个分散模块。

因此顶层路由语义应统一归属于刊播模块，不再让用户从导航层感知到旧模块割裂。

需要明确的是，这里的统一主要发生在前端模块认知、导航、页面结构与工作流层面；后端实现不要求同步收敛成一套路由命名。

允许继续保留按内容类型拆分的后端路由与 handler，例如：

- `post`
- `podcast`
- `video`

只要前端在刊播模块内完成统一组织，并且接口边界清楚、返回能力稳定即可。

### 10.2 创作页状态语义

创作页天然存在两种状态：

1. 当前频道总览态
2. 当前频道 + 当前合集的工作态

因此路由或页面状态模型必须能稳定表达：

- 只有频道
- 频道 + 合集

这能保证刷新、返回与从其他页面跳入时行为稳定。

### 10.3 切频道后的状态处理

如果用户切换频道：

1. 当前已选合集状态必须失效
2. 页面回到新频道的总览态
3. 发布按钮重新禁用
4. 等待用户在新频道中重新选择合集

不应保留旧频道合集状态，也不应自动映射到同名合集。

## 十一、错误处理与边界条件

### 11.1 没有频道

当用户进入刊播模块时，如果没有可用频道或当前频道已失效，应进入频道初始化状态，而不是显示空白页或跨频道总览。

初始化状态应提供：

1. 当前没有可用频道的说明
2. 创建或选择频道的入口
3. 刊播模块围绕频道展开的轻量解释

### 11.2 当前频道下没有合集

当用户已处于某频道，但该频道下还没有合集时：

#### 创作页

1. 合集层显示空状态
2. 内容层可继续展示该频道已有内容（若存在）
3. 发布按钮保持禁用
4. 明确提示先创建合集

#### 类型页

可提示全站暂无对应类型内容，并提供返回创作页的入口。

### 11.3 选错路径的防误导

以下行为必须避免：

1. 未选合集时允许直接发布
2. 文章合集下允许发布视频
3. 播客合集下允许进入文章编辑流
4. 视频合集工作态仍以文章 / 播客列表作为主视图

### 11.4 总览态与工作态边界

总览态允许按媒介自然混合呈现：

- 文章 + 播客线性流
- 视频卡片区

工作态必须聚焦：

- 文章 / 播客合集进入线性工作视图
- 视频合集进入卡片式视频工作视图

不应在工作态中继续把与当前合集无关的内容区块作为主内容。

## 十二、验证策略

本次设计落地后的验证重点不是“页面是否能打开”，而是“用户心智是否稳定成立”。

至少需要验证以下场景：

1. 点击 topbar 中的刊播，默认进入创作页
2. 进入创作页后，默认看到当前频道总览态
3. 切换频道后，是否正确回到新频道总览态
4. 旧频道下已选合集是否被正确清空
5. 未选合集时发布按钮是否被正确禁用并提供引导
6. 选中文章 / 播客 / 视频合集后，是否进入对应工作态
7. 频道总览页是否符合“文章+播客 5 条 + 视频卡片”的结构
8. 文章 / 播客 / 视频类型页是否作为全站探索入口，不受当前频道限制
9. 订阅 / 收藏是否明确表现为全局页且不随频道切换改变结果

工程验证要求：

1. 前端改动后运行 `bun run type-check`
2. 后端改动后运行 `go build ./...`
3. 若涉及路由或模块配置调整，需验证 topbar 顺序与页面标题是否符合设计

## 十三、结论

刊播模块是对博客、播客、视频三个一级模块的统一重构。

它的关键决策如下：

1. 合并后的统一模块名为 `刊播`
2. topbar 与页面标题全部使用功能名
3. topbar 前半段顺序固定为：订阅 → 刊播 → 音乐 → 论坛
4. 刊播默认进入 `创作` 页
5. 频道是上下文，合集是创作容器
6. 合集严格单类型：文章 / 播客 / 视频
7. 创作页默认展示频道总览态
8. 创作页中文章与播客在总览态下共用线性流，默认展示 5 条，可展开
9. 创作页中视频在总览态下单独以卡片区展示
10. 文章 / 播客 / 视频类型页是全站探索页，不受当前频道限制
11. 选中合集后进入对应的合集工作态
12. 订阅与收藏作为收在刊播模块中的全局能力页存在
13. 前端统一为刊播模块，不要求后端路由同步合并，`post` / `podcast` / `video` 等按类型拆分的后端边界可以继续保留

通过这套设计，可以在不强行统一底层内容模型的前提下，把三类内容整合成一个统一、稳定、可持续演进的刊播模块。
