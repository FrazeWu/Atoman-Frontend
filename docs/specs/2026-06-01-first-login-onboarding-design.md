# 首次登录引导设计

- Date: 2026-06-01
- Status: Draft for review
- Scope: 登录后一次性新手引导、模块介绍、Feed 订阅完成引导、账号级完成态

## 1. 背景

Atoman 目前已经具备登录、模块入口和 Feed 订阅能力，但新用户登录后没有统一的新手引导路径，用户需要自己理解各模块用途，再自行找到 Feed 入口完成首次订阅。

这会带来两个问题：

1. 新用户不知道每个模块是干什么的
2. 新用户不知道应该先去哪里完成一次订阅

本次设计要把这两个动作连成一个最小闭环：先介绍模块，再带用户完成一次真实订阅。

## 2. 目标

### 2.1 产品目标

- 让用户在第一次登录后快速理解 Atoman 的核心模块。
- 让用户在引导过程中实际完成一次订阅。
- 引导只对同一个账号出现一次。

### 2.2 体验目标

- 引导不打断登录后的正常进入节奏。
- 每一步都允许跳过。
- 引导结束后不再重复出现。

## 3. 非目标

本次设计不包含：

- 新的帮助中心
- 独立的新手教程页
- 多轮任务式教学
- 订阅能力本身的重构
- 模块权限或站点权限规则重写

## 4. 核心规则

### 4.1 触发时机

用户首次登录成功后，如果该账号的首次引导未完成，则自动展示引导。

触发点以账号状态为准，不以浏览器本地存储为准。

### 4.2 一次性规则

引导完成后，系统必须记录该账号已完成首次引导。

之后无论：

1. 更换设备
2. 更换浏览器
3. 清理本地缓存
4. 重新登录

都不应再次自动展示。

### 4.3 跳过规则

用户可以在任意一步选择跳过。

跳过后视为该账号已完成首次引导，不再再次出现。

## 5. 设计方案

### 5.1 推荐形态

本次采用“页面内高亮 + 逐步卡片”的引导形态。

原因：

1. 能直接指向真实入口，不需要用户在抽象教程和实际页面之间来回切换
2. 能自然串起“模块介绍”和“完成一次订阅”
3. 和当前站点已有的抽屉、纸张式 UI 风格一致

### 5.2 引导步骤

建议分为 4 步：

1. 站点总览：说明 Atoman 是一个由多个模块组成的平台
2. 模块介绍：用最短文案说明 Blog、Forum、Feed、Music、Debate、Timeline、Podcast、Video 各自作用
3. 进入 Feed：高亮 Feed 入口，提示用户去添加第一个订阅
4. 完成订阅：高亮 Feed 页的 `+ 订阅`，引导用户提交一次真实订阅

最后一步成功后，标记该账号首次引导完成。

## 6. 信息架构

### 6.1 模块介绍内容

模块介绍只需要一句短说明，不做长篇解释。

建议语义如下：

- Blog：发布文章、浏览频道、管理订阅关系
- Forum：发帖、回复、讨论话题
- Feed：聚合 RSS 订阅内容，阅读外部更新
- Music：音乐档案、歌曲、专辑与评论
- Debate：围绕议题进行辩论和投票
- Timeline：按时间浏览内容和记录
- Podcast：播客播放与节目内容
- Video：视频浏览、编辑与管理

### 6.2 完成订阅的定义

“完成一次订阅”指用户在 Feed 模块中成功提交一次有效订阅请求。

这里不要求：

1. 一定有内容更新
2. 一定完成分组整理
3. 一定阅读完文章

只要求真实创建出一条订阅记录。

## 7. 数据设计

### 7.1 账号级状态

在用户数据中增加一个一次性引导完成态字段，建议命名为：

- `onboarding_completed_at`

字段语义：

1. 为空：引导未完成
2. 非空：引导已完成

使用时间戳比布尔值更容易表达“完成时间”，也方便后续排查。

### 7.2 返回值扩展

登录和会话接口在返回当前用户信息时，需要带上首次引导完成态。

这样前端可以在登录后直接判断是否需要弹出引导，不需要额外再发一次查询。

## 8. 交互设计

### 8.1 展现方式

引导采用一个居中的浮层或抽屉式卡片叠层，覆盖当前页面但不离开当前站点。

每一步包含：

1. 标题
2. 简短说明
3. 当前步骤的下一步按钮
4. 跳过按钮

### 8.2 高亮规则

当步骤指向某个模块或入口时，需要高亮真实页面里的目标区域。

例如：

1. 模块介绍时，高亮模块入口区域
2. 进入 Feed 时，高亮 Feed 入口或 Feed 首页
3. 完成订阅时，高亮 `+ 订阅` 按钮和订阅抽屉

### 8.3 结束规则

满足以下任一条件即结束引导并写入完成态：

1. 用户完成最后一步订阅
2. 用户点击跳过

## 9. 实现边界

### 9.1 后端

后端只负责：

1. 保存首次引导完成态
2. 在登录/会话返回中携带该状态

不新增独立的 onboarding API 页面，不做复杂状态机。

### 9.2 前端

前端只负责：

1. 首次登录时判断是否需要展示
2. 展示步骤卡片和高亮
3. 引导用户进入 Feed 并完成一次订阅
4. 在完成或跳过后立即关闭引导

### 9.3 订阅页联动

Feed 页面已经有 `+ 订阅` 抽屉能力，本次只需要把引导和该入口串起来，不重做订阅表单。

## 10. 错误处理

1. 如果引导状态保存失败，优先保证订阅或跳过动作本身完成
2. 如果状态读取失败，默认不展示引导，避免阻塞正常登录
3. 如果跳转到 Feed 失败，给出简单提示并允许用户手动继续

## 11. 测试策略

最小验证即可，重点看这 4 件事：

1. 新账号首次登录会展示引导
2. 同账号再次登录不会展示引导
3. 任意一步跳过后不再展示
4. 完成一次真实订阅后引导结束并被记录

## 12. 不做的事

本次明确不做：

1. 多语言引导
2. 可配置的引导文案后台
3. 新手任务积分
4. 多次循环教学
5. 独立帮助中心页面

## 13. 结论

本次首次登录引导采用“账号级一次性、逐步引导、最终完成一次 Feed 订阅”的最小方案。

它的核心价值只有两个：

1. 让用户知道各模块是干什么的
2. 让用户真正完成一次订阅

并且只出现一次。
